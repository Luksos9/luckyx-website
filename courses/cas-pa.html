2.The formula for the "% of incidents resolved by first assigned group" is configured as: ([[Number of resolved incidents by first assigned group]] / [[Number of resolved incidents]]) * 100. You observe a "No Score" output when the denominator indicator equals 0. What is the best method to enhance this formula so it returns 0 instead of "No Score"?
  • AEnable the property "Allow formula component to be NULL".
  • BSet the system property "Value when nil" to 0.
  • CAdd a conditional JavaScript statement in the formula to return 0 if the value of Number of Resolved Incidents is 0.
  • DEnable the property "Render continuous lines".
Show full explanation
Correct Answer: Add a conditional JavaScript statement in the formula to return 0 if the value of Number of Resolved Incidents is 0.SourceOfficial Doc: Formula IndicatorsIf the link fails, search Google for: ServiceNow formula indicator division by zero JavaScript conditionalExpert ExplanationFormula Indicators support JavaScript expressions. When a denominator indicator equals zero, the formula produces a "No Score" result. The correct approach is to add a conditional check directly in the formula: if ([[denominator]] == 0) { 0 } else { ([[numerator]] / [[denominator]]) * 100 }. This ensures the formula gracefully returns 0 instead of failing.

Why the Others Are WrongAllow formula component to be NULL: This is not a real property that addresses division-by-zero.Value when nil: Not a system property that resolves formula calculation errors.Render continuous lines: A widget display setting that draws lines between gaps in data. It does not change how the formula calculates.

Memory Tip"Guard the denominator with JavaScript." Formula Indicators use JavaScript, so standard coding practices like null/zero checks apply.

Real-World ExampleOn a weekend when no incidents are resolved, the denominator "Number of Resolved Incidents" drops to 0. Without the JavaScript guard, the scorecard shows "No Score." With the conditional, it cleanly shows 0%, maintaining the trend line.

  • DAt least 12 months

Correct Answer: At least 12 monthsSourceOfficial Doc: ForecastingIf the link fails, search Google for: ServiceNow Performance Analytics forecasting minimum historical data 12 monthsExpert ExplanationPerformance Analytics forecasting uses algorithms that analyze historical patterns to project future trends. 5.When configuring a Bucket Group for a continuous variable like 'Incident Age (Days)', what happens to records whose age exceeds the maximum value defined in the highest available bucket during data collection?
  • AThe entire data collection job aborts immediately.
  • BThe record is automatically assigned to a newly generated dynamic bucket.
  • CThe record is processed but placed into the 'Unassigned' or empty breakdown element category.
  • DThe system automatically alters the maximum value of the highest bucket to accommodate the record.
Show full explanation
Correct Answer: The record is processed but placed into the 'Unassigned' or empty breakdown element category. Source Official Doc: Bucket Groups If the link fails, search Google for: ServiceNow Performance Analytics Bucket Group out of range unassigned element Expert Explanation Bucket Groups convert continuous numeric fields into discrete breakdown elements for analysis. When a record's value (e.g., incident age = 95 days) does not fit within any defined bucket range, the data collection engine assigns that record to an unassigned breakdown element rather than discarding the record or aborting the job. This signals the administrator that a catch-all bucket (such as "91 days or more") should be added to the Bucket Group configuration.

Why the Others Are Wrong Job aborts immediately: The collection job is resilient to out-of-range values. It continues processing all remaining records and simply leaves the unassigned count visible for review. Assigned to a dynamic bucket: Bucket ranges are statically defined. The system does not create dynamic or overflow buckets during collection. System alters the maximum value: Bucket boundaries are never modified automatically. Administrators must explicitly update the Bucket Group to extend the range.

Memory Tip No bucket? Goes to lost-and-found (Unassigned). Seeing a high Unassigned count on a bucket-based breakdown means your highest bucket ceiling is too low. Add an open-ended bucket. UI Navigation Path: Performance Analytics > Breakdowns > [Breakdown] > Bucket Groups related list What to look for: The list of defined bucket ranges. Add a final row with a high upper limit (e.g., 999999) to create a catch-all bucket for extreme values.

Real-World Example An Incident Age breakdown has buckets: 0-7 days, 8-14 days, 15-30 days, 31-60 days. An incident aged 90 days falls into Unassigned. After reviewing the dashboard and seeing 15% of incidents in Unassigned, the administrator adds a "61+ days" bucket with an upper range of 9999 to capture all long-running incidents correctly.

  • FThe Widget is configured to follow an Interactive Filter.
Show full explanation
Correct Answers: The Indicator has no configured Breakdowns, "Follow Element" is not checked, and Element security prevents data access.SourceOfficial Doc: Dashboard Breakdown SourceIf the link fails, search Google for: ServiceNow PA widget not following dashboard breakdown source follow elementExpert ExplanationA PA widget may not follow the Dashboard Breakdown Source for three reasons. Why the Others Are WrongTime series type widget: Time series widgets support breakdown filtering. The widget type is not a barrier to following the Dashboard Breakdown Source.Different Breakdown

  • C2 - Two levels of Breakdowns are always collected by default.

Correct Answer: 1: Without the matrix, a single level of Breakdowns is collected.SourceOfficial Doc: Automated IndicatorsIf the link fails, search Google for: ServiceNow collect breakdown matrix single level two levelsExpert ExplanationThe Collect breakdown matrix checkbox on the Indicator record controls whether intersecting data across multiple breakdown dimensions is collected.
