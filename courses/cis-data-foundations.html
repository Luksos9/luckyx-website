3.The Staleness metric calculation is based on which of the following fields?
  • AMost recent discovery <last_discovered>
  • BCreated <sys_created_on>
  • CUpdated <sys_update_on>
  • DFirst Discovered <first_discovered>
  • EDiscovery Source <discovery_source>
Overall explanation:The Staleness metric calculation is based on the Updated <sys_update_on> field.A Configuration Item (CI) is considered stale if it has not been updated in the CMDB for a specified period of time. The CMDB calculates this period by checking the value of the sys_updated_on field. This field reflects the timestamp of the last modification made to the CI record. Importantly, even if an automated tool like ServiceNow Discovery scans a CI and finds no attribute changes, the act of that scan updates the sys_updated_on timestamp, thereby keeping the CI from being flagged as stale.

13.Aligning to the Common Service Data Model (CSDM) allows for auto-population of the assignment group field on new incident and change request records. Which of the following are true when creating a new Incident or Change Request in ServiceNow? (Choose 2) (choose 2)
  • AChange group on the selected CI auto-populates the assignment group field on the change request form.
  • BApproval group on the selected CI auto-populates the assignment group field on the change request form.
  • CSupport group on the selected CI auto-populates the assignment group field on the incident form.
  • DManaged by group on the selected CI auto-populates the assignment group field on the incident form.
The correct statements regarding the auto-population of the Assignment Group field on Incident and Change Request records when aligning with the Common Service Data Model (CSDM) are:Change group on the selected CI auto-populates the assignment group field on the change request form.Support group on the selected CI auto-populates the assignment group field on the incident form.ExplanationAligning Configuration Items (CIs) with CSDM and populating key non-discoverable group fields enables the automation of ITSM processes, particularly routing Incidents and Changes.Incident Routing (Support Group):The Support Group attribute configured on a CI record (or a service offering) is used to determine Incident routing.A CI, when selected from the incident form, has its support group value auto-populate the assignment group field on the incident form to help streamline incident routing.In a lab scenario, selecting a Service Offering configured with a Support Group caused the Assignment Group field on the incident record to auto populate.Change Routing (Change Group):The Change Group attribute configured on a CI record (or a service offering) maps to the Change Assignment Group.The Change group on the selected CI auto-populates the assignment group field on the change request form.This auto-population streamlines the change process by dynamically routing change requests to the appropriate group.Note on Precedence: The system checks for the appropriate group assignment in a specific order:If a group attribute (like Support Group or Change Group) is assigned directly to the Configuration Item (CI), that value is used.If the CI level field is not populated, the system takes the assignment group defined at the corresponding Offering level.The Managed by Group is primarily used to streamline the assignment of Data Manager tasks for attestation, life cycle, and certification policies, not the primary assignment group for general Incidents or Changes.

  • AOperational status
  • BInstall Status
  • CHardware Status
  • DAsset tag
Based on the sources, the two fields that are automatically synchronized on the associated Configuration Item (CI) record when the State field is updated on an Asset record are Install Status and Hardware Status.The correct choices are:Install StatusHardware StatusExplanationThe synchronization of life cycle fields ensures that the status of assets and CIs remain aligned.Legacy Life Cycle Fields: The State field on the Asset record, along with the Install Status and Hardware Status fields on the CI record, are referred to as legacy life cycle fields that the base system includes automation to keep synchronized.Synchronization Mechanism: The asset State field, when updated, is synchronized with the CI Install Status field.Confirmed Synchronization: The source material confirms that when the State field is updated on an Asset record, both the Install Status and Hardware Status fields are automatically synchronized on the associated CI record.It is worth noting that this synchronization is often directional; for example, if the CI Install Status field is changed, it is generally not synchronized back to the Asset State field.The specific field that updates depends on the CI class.
