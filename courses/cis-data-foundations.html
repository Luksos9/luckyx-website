1.Which results in a reclassification task being generated?
  • Aglide.class.upgrade.enabled set to false and during discovery a Server is classified as a Windows server.
  • Bglide.class.switch.enabled set to true and during discovery a Windows server is classified as a Linux server.
Show full explanation
The correct option is:glide.class.upgrade.enabled set to false and during discovery a Server is classified as a Windows server.ExplanationA reclassification task is generated when an automatic reclassification is BLOCKED (set to false).Scenario 1: Server classified as Windows Server (Upgrade)Reclassifying a Configuration Item (CI) from the general Server [cmdb_ci_server] class to the more specific Windows Server [cmdb_ci_win_server] class is defined as a class Upgrade.The system property governing this automation is glide.class.upgrade.enabled.When this property is set to false, the reclassification cannot take place automatically, and instead, the system generates a reclassification task for manual review.Scenario 2: Windows Server classified as Linux Server (Switch)Reclassifying a CI from the Windows Server class to the Linux Server class is defined as a class Switch, as they are in different branches of the hierarchy.The scenario states glide.class.switch.enabled is set to true. When automatic reclassification properties (glide.class.upgrade.enabled, glide.class.downgrade.enabled, glide.class.switch.enabled) are set to true, reclassification happens automatically, and no reclassification task is created.

6.What is the default threshold value that the IRE uses to determine when it should stop updating the oldest duplicate CI record, thereby skipping updates on all records when that threshold is exceeded?
  • A2
  • B3
  • C5
  • D10
Show full explanation
The default threshold value that the Identification and Reconciliation Engine (IRE) uses to determine when it should stop updating the oldest duplicate CI record is 5.ExplanationThis threshold is controlled by a specific system property designed to manage the behavior of the IRE when encountering multiple duplicate Configuration Items (CIs) in the CMDB.The system property responsible for this threshold is glide.identification_engine.skip_duplicates.threshold.This property is set to 5 by default.When the property glide.identification_engine.skip_duplicates is set to true (the default state), and the number of duplicate CIs is less than the threshold (5), the IRE selects the oldest CI record (based on creation date) as a match and updates it. The other duplicate CIs are tagged using the Duplicate of field.However, if the number of duplicate CIs exceeds this threshold (e.g., if 6 duplicate CIs exist and the threshold is 5), the IRE will not update any of the records. The IRE prevents the update, and if new data comes in, a new de-duplication task is created, but none of the existing duplicate records are updated.The sources provide an example illustrating this: if the threshold is set to 3, and a 4th duplicate CI is added, the update behavior is stopped, and none of the records are updated.

9.When configuring an Identification Rule entry, what is the impact of checking the Allow fallback to parent's rules option?
  • AIt allows null attribute values for that identifier entry.
  • BThe system uses the parent's reconciliation rule priority if the child rule fails.
  • CIf the current CI's identifier entry fails to find a match, the IRE attempts matching using the criterion attributes of the parent class.
  • DIt prevents the CI from being reclassified.
Show full explanation
The correct impact of checking the Allow fallback to parent's rules option when configuring an Identification Rule entry is:If the current CI's identifier entry fails to find a match, the IRE attempts matching using the criterion attributes of the parent class.ExplanationThe Identification and Reconciliation Engine (IRE) uses Identification Rules to uniquely identify Configuration Items (CIs). These rules are defined by one or more identifier entries, which specify a unique set of criterion attributes.The sources specifically describe the function of the Allow fallback to parent's rules option:The Allow fallback to parent's rules checkbox is an option within the identifier entry configuration.When this option is checked, if the Identification Rule associated with the current CI class fails to find a match using its configured identifier entry criteria, the IRE will then proceed to the parent CI class identification rule and attempt to find a match there. This ensures a match is attempted using the broader criteria defined by the parent class if the child class's specialized rule fails.In contrast:Allow null attributes is a separate checkbox option. When checked, it allows matching to be attempted even if some criterion attributes are null, provided at least one attribute is not null. 12.Which of the following can be configured to prevent inserts into the CMDB from an unwanted data source?
  • AIdentification Rule
  • BReconciliation Rule
  • CDynamic Reconciliation Rule
  • DData Refresh Rule
  • EIRE Data Source Rule
Show full explanation
The correct configuration used to prevent inserts into the Configuration Management Database (CMDB) from an unwanted data source is the IRE Data Source Rule.Overall Explanation:The overall process of bringing data into the CMDB is governed by the Identification and Reconciliation Engine (IRE). While many rules focus on managing updates (reconciliation) or preventing duplicates (identification) of existing records, a specific rule is required to block the creation of new records (inserts) entirely from a designated source.IRE Data Source Rules are specifically configured to block inserts from a specific data source into a specified class in the CMDB. They can be configured by navigating to Configuration > Identification/Reconciliation > IRE Data Source Rules.

Why the other options are incorrect (focusing on updates/matches, not inserts):Identification Rule: Identification rules are crucial for ensuring data integrity by uniquely identifying CIs and preventing the creation of duplicate records. They determine if an incoming CI is new (leading to an insert) or existing (leading to an update). They do not function to block the insert action based on the source itself.Reconciliation Rule (Static Reconciliation Rule): These rules are designed to manage and prioritize updates to existing CIs by specifying which data sources are authorized to modify CI attributes and resolve conflicts between sources. The reconciliation process is generally not part of the creation (insert) process of a CI; it follows identification.Dynamic Reconciliation Rule: These rules handle complex data reconciliation scenarios for updates to existing CIs by leveraging Multisource CMDB data to select values (e.g., Largest Value, Most Reported). Like static rules, they primarily govern how existing records are updated, not whether a new record (insert) is permitted.Data Refresh Rule: These rules are configured under the Reconciliation Rules tab and manage how and when CI data can be refreshed if a higher-priority data source has failed to update the record within a specified time frame (staleness). They manage updates and data staleness, not inserts.

14.The following Reconciliation Rules were configured for ServiceNow, Altiris, and SCCM for the Windows Server (cmdb_ci_win_server) class: ServiceNow: Priority 100 Altiris: Priority 200 SCCM: Priority 300 Which statements are true? (Choose 2) (choose 2)
  • AData collected with a discovery source of Altiris can update records inserted by SCCM into the Windows Server table.
  • BData collected with a discovery source of ServiceNow can insert new records into the Windows Server table, but cannot update records created by Altiris or SCCM.
  • CData collected with a discovery source of SCCM can be inserted as new records in the Windows Server table.
  • DData collected with a discovery source of SCCM can update any record in the Windows Server table because it has the highest priority number.
Show full explanation
Correct Answer: Data collected with a discovery source of Altiris can update records inserted by SCCM into the Windows Server table; Data collected with a discovery source of SCCM can be inserted as new records in the Windows Server table Source Official Doc: Understanding IRE Reconciliation Rules If the link fails, search Google for: ServiceNow CMDB reconciliation rules priority data source precedence update insert Expert Explanation In ServiceNow reconciliation, lower priority number = higher authority (Priority 100 outranks Priority 200). Two key rules govern data source behavior: (a) a higher-priority source can update records from a lower-priority source, and (b) any authorized source can insert new records regardless of its priority. With ServiceNow at 100, Altiris at 200, and SCCM at 300: Altiris (200) can update SCCM records (300) because 200 outranks 300. SCCM (300) can insert new records because insertion is unrestricted by priority. SCCM cannot update records from ServiceNow or Altiris because it has the lowest priority.

Why the Others Are Wrong ServiceNow can insert but cannot update: ServiceNow has Priority 100 (highest authority). It can both insert new records and update any existing record from any source. There is no restriction on the highest-priority source. SCCM can update any record because it has the highest priority number: Higher priority number means lower authority. SCCM at 300 has the lowest authority and can only update its own records, not those from ServiceNow (100) or Altiris (200).

Memory Tip Lower number = higher boss. Any source can hire (insert), but only a higher boss can overwrite (update) a lower one's work.

Real-World Example Altiris discovers a Windows Server and sets its location to "Building A." SCCM later discovers the same server and tries to set location to "Building B." Since SCCM (300) has lower priority than Altiris (200), the update is rejected and "Building A" remains. But when SCCM discovers a brand-new server not in the CMDB, it inserts the record successfully because insertions are unrestricted.
