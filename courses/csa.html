1.When importing data, what happens to imported rows if no coalesce field is specified?
  • AAll rows are rejected; coalesce is required.
  • BAll rows are treated as new records. No existing records are updated.
  • CDuplicate rows are rejected from the import.
  • DAll rows are treated as new records, but errors will be flagged.
Show full explanation
Correct Answer: All rows are treated as new records. No existing records are updated.SourceOfficial Doc: Coalesce ValuesIf the link fails, search Google for: ServiceNow Zurich coalesce field import transform no coalesce new recordsExpert ExplanationThe coalesce field tells the transform map how to match incoming rows to existing records. Without it, there is no matching logic, so the transform treats every row as new and inserts it. This means running the same import twice without coalesce creates duplicate records. The import does not fail or flag errors for missing coalesce.

Why the Others Are WrongAll rows rejected: Coalesce is optional. Without it, imports proceed and create new records.Duplicate rows rejected: No duplicate detection occurs without coalesce. All rows are inserted.New records with errors flagged: No errors or warnings are generated for missing coalesce fields.

Memory TipNo coalesce = no matching = all new records. The transform has no way to find existing records to update.

Real-World ExampleYou import 500 employee records without coalesce. Running the same import again creates 500 duplicates because the transform cannot identify that those employees already exist.

3.What field contains a record’s 32-character, unique identifier?

Show full explanation
Correct Answer: sys_idSourceOfficial Doc: System DictionaryIf the link fails, search Google for: ServiceNow Zurich sys_id field 32 character unique identifier recordExpert ExplanationThe sys_id field stores the unique 32-character GUID (Globally Unique Identifier) for every record in ServiceNow. Why the Others Are Wrongsn_rec_id, rec_id, u_id, sn_gu_id, sn_sys_id: None of these are real ServiceNow system field names. They are all fabricated to test whether you know the correct field name: sys_id.

6.How to define an ACL to allow itil users to create incidents?
  • AName: incident.None; Operation: create; Role: itil
  • BName: incident.Any; Operation: write; Permission: itil
  • CName: incident:*; Permission: write; Role: itil
  • DName: incident.None; Permission: create; Role: itil
  • EName: incident:*;Operation: write; Permission: itil
Show full explanation
Correct Answer: Name: incident.None; Operation: create; Role: itilSourceOfficial Doc: Access Control RulesIf the link fails, search Google for: ServiceNow Zurich ACL incident create operation itil role table-levelExpert ExplanationTo define an ACL allowing itil users to create incidents: set Name to incident.None (table-level with no specific field), Operation to create, and add Role: itil. The "None" in the name indicates a table-level ACL rather than a field-level one. The Operation field specifies the permitted action.

Why the Others Are Wrongincident.Any; Operation: write: "write" covers updates, not record creation. The operation must be "create."incident:*; Permission: write: Wildcard (*) and colon (:) are not standard ACL naming. Uses "Permission" instead of "Role."incident.None; Permission: create: Incorrectly labels the create value as "Permission" instead of "Operation."incident:*; Operation: write: Wildcard naming is incorrect, and "write" is the wrong operation for creation.

Memory TipACL format: table.field; Operation; Role. Use "None" for table-level, "create" for new records, and the required role.

Remove duplicate, obsolete, or inaccurate records from the source file before loading. 12.Create Incident, Password Reset, and Report outage: what do these services in the Service Catalog have in common?
  • AThey direct the user to a record producer
  • BThey direct the user to a catalog property
  • CThey direct the user to a catalog UI policy
  • DThey direct the user to a catalog client script
Show full explanation
Correct Answer: They direct the user to a record producerSource Official Doc: Record Producers If the link fails, search Google for: ServiceNow Zurich record producer create incident password reset report outageExpert ExplanationThese are all Record Producers in the Service Catalog. "Create Incident" produces an Incident record, "Password Reset" produces a password reset request record, and "Report Outage" produces an outage record. Each one uses the Record Producer mechanism to create records directly on their target tables from a catalog-style form.

Why the Others Are Wrong Catalog property: Catalog properties are configuration settings (like approval rules). They are not where users are directed. Catalog UI policy: UI policies control field behavior on catalog forms (show/hide, mandatory). They are not the service type itself. Catalog client script: Client scripts run browser-side logic. The services themselves are Record Producers, not scripts.

Memory TipIf a catalog item creates a record on a specific table, it is a Record Producer. Create Incident = record on incident table. Password Reset = record on password reset table.

Real-World ExampleAn employee clicks "Report an Outage" in the Service Catalog. Behind the scenes, this Record Producer creates a record directly on the Outage table with the details the user provided, bypassing the standard REQ/RITM process.
13.When building a table extended from a base table, which fields do you need to create? (Choose two.) (choose 2)
  • AThe fields that are not in the base table
  • BThe mandatory fields for the base table
  • CThe fields that are specific to the extended table
  • DThe reference fields for the base table
Show full explanation
Correct Answers: The fields that are not in the base table; The fields that are specific to the extended tableSourceOfficial Doc: Table ExtensionIf the link fails, search Google for: ServiceNow Zurich table extension inherited fields create new fieldsExpert ExplanationExtended tables inherit all fields from their parent/base table. You only need to create fields that are new and specific to the extended table. Both correct answers express the same concept: only create what does not already exist in the base table. For example, Incident inherits Number, Short description, and State from Task, then adds its own Category field.

Why the Others Are WrongMandatory fields for the base table: Already inherited. Recreating them would cause conflicts.Reference fields for the base table: Also inherited. All field types from the base table carry over to extensions.

Memory TipInheritance means you get everything for free. Only build what is new and unique to your extended table.
