Source: DocumentationThe correct answer is: Add each set of duplicate CIs to a de-duplication task.In the documentation, it's stated that when the identification process encounters duplicate CIs, it groups each set of duplicate CIs into a de-duplication task for review and remediation.

Explanation of other options:Send a notification to the CI owner: While this could be part of a broader incident management strategy, it isn't the default behavior described in the provided documentation for handling duplicate CIs during the Identification and Reconciliation phase.Automatically create an associated identification rule: Again, while creating or updating identification rules may be part of a strategy to prevent duplicate CIs, it's not the default action for handling detected duplicates.Stop the next discovery for the CI that is duplicated: This is not mentioned in the documentation. 2.What must be installed on the MID Server to enable credential-less Discovery?
  • ANMAP
  • BPort Scanner plugin
  • CWindows PowerShell module
  • DOpenSSH server
Show full explanation
Correct Answer: NMAP Source ServiceNow KB0970202: Credential-less Discovery setup If the link fails, search Google for: ServiceNow Zurich credential-less Discovery NMAP MID Server installation KB0970202 Expert Explanation Credential-less Discovery (also called agentless scanning without credentials) uses NMAP (Network Mapper) installed on the MID Server host to perform port scanning and OS fingerprinting without requiring login credentials on the target hosts. According to official documentation: "If you decide to use credential-less Discovery, you must install Nmap on each Windows MID Server." NMAP performs TCP/UDP port scans, banner grabbing, and OS detection to populate CMDB with basic host information even when no credentials are available.

Why the Others Are Wrong Port Scanner plugin: No such ServiceNow plugin exists for the MID Server. NMAP itself is the scanning engine, installed as a standalone tool on the MID Server's operating system. PowerShell module: PowerShell enables credential-based Windows discovery (WMI queries). It has no role in credential-less discovery, which by definition avoids authenticating to target systems. OpenSSH server: OpenSSH is an SSH daemon on target Linux servers. It is the target-side component that accepts SSH connections, not a tool installed on the MID Server for credential-less scanning.

Memory Tip No credentials? You need a scanner. NMAP is THE network scanner = install it on the MID Server.

Real-World Example A company wants to build an initial CMDB inventory of a legacy network segment where no credentials are available. The admin installs NMAP 7.94 on the Windows MID Server host, configures Discovery to use credential-less mode for the 10.50.0.0/16 range, and runs Discovery. NMAP scans all 65,535 hosts, identifying active devices by open ports and OS fingerprinting, populating the CMDB with IP, MAC, OS type, and open port information without ever logging into a single target.

9.What cannot be configured within a ServiceNow Discovery Schedule?
  • ACredentials
  • BIP Range Sets
  • CMID Server selection
  • DDiscovery schedule frequency (run interval)
Show full explanation
Correct Answer: Credentials Source ServiceNow Community: Best practices for setting up Discovery schedules If the link fails, search Google for: ServiceNow Zurich Discovery Schedule configuration options credentials excluded Expert Explanation A Discovery Schedule defines when and where Discovery runs: the run frequency, IP Range Sets, MID Server, Behaviors (which probes to launch), Location tag, and Shazzam batch size. Credentials are explicitly not configured within a Discovery Schedule. Instead, they live in a separate Discovery > Credentials module and are matched to targets at runtime through Credential Affinity Rules (which map credential types to IP ranges or CI attributes). This separation allows the same credential to be reused across multiple schedules and eliminates the need to update every schedule when a password changes.

Why the Others Are Wrong IP Range Sets: These are a core element of every Discovery Schedule, defining which addresses to scan. They are absolutely configurable within the schedule. MID Server: You select which MID Server (or MID Server cluster) handles this schedule's probes directly in the schedule form. Schedule frequency: The run frequency (daily, weekly, monthly, or on-demand) is one of the first things configured when creating a Discovery Schedule.

Memory Tip Credentials live in their own module, not in schedules. Think of it as: the schedule tells WHERE and WHEN to look; credentials tell HOW to log in. They are stored separately. UI Navigation Path: All > Discovery > Discovery Schedules > [select any schedule] What to look for: The schedule form shows IP Ranges, MID Server, Behaviors, and Schedule tabs. Notice there is no Credentials tab. Credentials are managed at: All > Discovery > Credentials

Real-World Example An organization has 5 Discovery Schedules covering different network segments. The Windows admin account password is rotated monthly. Because credentials are centralized in Discovery > Credentials rather than embedded in each schedule, only one record needs updating. All 5 schedules automatically use the new password on their next run without any schedule changes.

14.Which of the following are valid syntaxes for referencing a tabular variable in a ServiceNow Discovery pattern step? (Select 3) (choose 3)
  • A$table[1].example
  • B$table[].example
  • C$table[*].example
  • D$table[X].example
Show full explanation
Correct Answers: $table[1].example, $table[].example, and $table[*].example Source ServiceNow Community: Tabular variable syntax in patterns If the link fails, search Google for: ServiceNow Zurich Discovery pattern tabular variable syntax NDL table wildcard index Expert Explanation Tabular variables in ServiceNow Discovery patterns (NDL) use bracket notation to access rows. The three valid syntaxes serve different purposes: $table[1].field accesses a specific row by 1-based index (row 1 is the first row). $table[].field is used in iteration contexts, processing the current row during a loop. $table[*].field collects the specified field value from every row simultaneously, returning a multi-value result. These map directly to the tabular variable interaction options in Pattern Designer step configuration.

Why the Others Are Wrong $table[X].example: "X" is not a defined token in NDL. It is not a wildcard (that is [*]), not an iterator (that is []), and not a number. Using it would cause a pattern syntax error.

Memory Tip [1] = pick one row by number. [] = walk through rows one by one. [*] = grab all rows at once. The absent X has no meaning in this language.

Real-World Example A pattern returns a table of running processes. Using $processes[*].pid retrieves all PIDs from every row for bulk evaluation. Using $processes[1].name gets only the first process name. Using $processes[].port iterates through each row in a loop to check ports one by one. All three syntaxes serve different use cases within the same pattern.
